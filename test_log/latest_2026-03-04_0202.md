# Performance-First Hypothesis Set (2026-03-04)

## 1) Optimization Principle
- Primary objective: maximize throughput and reduce tail latency (`p95/p99`) even if UX convenience is reduced.
- Decision priority:
1. timeout rate
2. p99
3. p95
4. success rate
5. p50

## 2) Pipeline Decomposition
1. Client-side image acquisition/serialization
2. Network handshake + upload transfer
3. Backend request parsing/normalization
4. Gemini upstream inference call
5. Response parsing and return
6. Runtime effects (cold start, concurrency saturation, failover path)

## 3) Performance Hypotheses (ranked)

### H1. Cold-start dominates worst-case latency
- Mechanism: Render instance wake-up and dependency cold path inflate first request.
- Expected symptom: first request after idle >> steady-state p50.
- Metric: `cold_start_delta = cold_time - warm_p50`.
- Falsification: if cold_start_delta < 2s in repeated cycles.

### H2. Gemini upstream dominates end-to-end time
- Mechanism: model response latency larger than local preprocessing/network.
- Metric: backend internal split (`preprocess_ms`, `gemini_call_ms`, `postprocess_ms`).
- Falsification: if `gemini_call_ms < 40%` of total on median runs.

### H3. Image normalization reduces invalid input failures but adds small fixed cost
- Mechanism: preprocess adds CPU time but prevents upstream image parse errors.
- Metric: compare norm-on vs norm-off:
- invalid_argument_rate
- p95/p99
- timeout rate
- Falsification: norm-on has equal/higher failure and worse tail by significant margin.

### H4. Tail latency is sensitive to large JPG path under load
- Mechanism: bigger binary + normalization + model processing creates queueing.
- Metric: per-file p95/p99 under concurrency 4/8.
- Falsification: large JPG p95 within 10% of smallest PNG across loads.

### H5. Concurrency saturation begins before level 8
- Mechanism: shared CPU/network/upstream quota causes queue buildup.
- Metric: latency slope and timeout growth from concurrency 1->2->4->8.
- Falsification: p95 growth nearly linear and timeout remains <2% at 8.

### H6. Key failover path adds recovery latency spikes
- Mechanism: first key failure + retry/switch sequence introduces outlier delay.
- Metric: requests near switch event vs steady-state p95.
- Falsification: no measurable outlier around switch logs.

### H7. Request body size inflation increases upload + parse overhead
- Mechanism: larger bodies (and potential base64 segments upstream) increase transfer and parsing time.
- Metric: `request_bytes` vs `time_total` correlation.
- Falsification: weak/no correlation in controlled runs.

### H8. Network handshake reuse is inconsistent
- Mechanism: lack of connection reuse increases handshake cost.
- Metric: client-side connect/tls timings and keep-alive reuse rate.
- Falsification: handshake cost consistently negligible (<5% total).

### H9. Timeout budget (45s) hides two distinct failure classes
- Mechanism: same timeout captures cold-start stalls and true overload stalls.
- Metric: classify timeout by phase (early stall vs late stall).
- Falsification: single homogeneous timeout pattern.

### H10. Parsing/serialization cost is non-trivial at tail
- Mechanism: JSON parse/encode and multipart boundary handling accumulate under load.
- Metric: backend phase timers around parse/response serialization.
- Falsification: parse+serialize consistently <5% total even at concurrency 8.

### H11. Norm-off branch has slightly better median but worse reliability tail
- Mechanism: skipping preprocess saves median time but increases invalid input/timeouts.
- Metric: A/B delta (`p50`, `p95`, timeout rate, success).
- Falsification: norm-off beats norm-on on all four metrics.

### H12. Service-level variance contributes more than image variance at p99
- Mechanism: runtime variance (instance state, quota state) dominates file-specific differences.
- Metric: same file repeated 30x vs mixed-file run variance.
- Falsification: per-file variance always smaller than cross-file variance.

### H13. Retry/failover policy shifts errors to latency outliers
- Mechanism: retries reduce hard failures but convert them into long-tail responses.
- Metric: compare no-retry/failover-off vs current policy in controlled window.
- Falsification: retries reduce both failures and tail with no outlier increase.

### H14. Throughput ceiling is upstream-model bound, not ingress bound
- Mechanism: backend ingress handles bytes faster than Gemini can respond.
- Metric: upload completion time vs total completion time gap.
- Falsification: upload phase dominates total time in p95/p99.

### H15. Performance-optimal operating mode is warmed, low-moderate concurrency
- Mechanism: warm instances + controlled in-flight requests avoid queue collapse.
- Metric: best efficiency point on concurrency-response curve.
- Falsification: highest concurrency still has stable p95/p99 and low timeout.

## 6) Decision Rules (strict, performance-first)
1. Choose candidate with lower timeout rate first.
2. If timeout tie, choose lower p99.
3. If p99 tie, choose lower p95.
4. If p95 tie, choose higher success rate.
5. Median (`p50`) is tie-breaker only.
